2. How the LLM Gets the API Key

The LLM itself does not store the key the MCP client (the environment where the LLM is running) does.

For example:

  • In ChatGPT Enterprise (with MCP support), you register your MCP server in the settings and paste in:

    • The Manifest URL (https://mcp.actus.tv/mcp-manifest.json)
    • The API Key (Bearer xxxxxxxx)

  • The MCP client stores the key locally and attaches it to every HTTPS call it makes to your server.

  • The model never sees the raw key unless you explicitly return it in your tool responses (which you should never do).

So: You give the API key once during MCP server registration in the LLM client.

Yes I mean your API key.

How rotation works:

  • You generate a new key on your server.

Theres no automatic key update you manually update it wherever the MCP client is configured.

2. Which MCP Does the LLM Choose if Multiple Servers Offer Similar Tools?

If the LLM is connected to multiple MCP servers with overlapping tool names/descriptions (e.g., both Actus and another company offer get_transcripts), the choice depends on:

  1. Tool Descriptions

    • The LLM compares the descriptions in each tools_schema.json to the user query.
    • If your description matches the users intent more closely (TV and radio broadcast transcripts from monitored channels) vs. a generic one, the LLM will prefer yours.

  2. Registration Priority

    • In ChatGPT and Claude, the MCP client (users configuration) decides which MCPs are active.
    • If the user only registers your MCP youre the only option.
    • If they register two the model decides based on semantic matching.

  3. Parameter Fit

    • If your tool parameters fit the query exactly (e.g., you have channel, start_time), the LLM tends to choose yours.

  4. Name Collision Rules

    • If two tools have exact same name, some MCP clients might namespace them internally (e.g., actus.get_transcripts vs other.get_transcripts).
    • Then the LLM can call either explicitly.

Bottom line
The LLM does not go out to the open internet to pick a provider it only calls tools for MCPs that are already registered in the clients settings.
If two MCPs are registered, selection is based on semantic matching of tool descriptions to the users request.

1. VPN Approach How AWS Sees Your Local MongoDB

The core idea

A VPN (Virtual Private Network) is like running a long, secure network cable between your AWS server and your offices LAN.
Once connected, AWS can see your local machines private IPs (like 10.x.x.x or 192.168.x.x) as if it were physically inside your building.

How the AWS MCP server accesses mongodb://10.x.x.x:27017

  1. Your company network probably already has private IP addresses for internal machines (MongoDB server is one of them).

  2. You set up a VPN gateway in AWS and a VPN gateway on your office network (this might be on a firewall/router or a dedicated box).

  3. The gateways authenticate to each other and encrypt all traffic between them.

  4. Once connected:

    • AWS gets a route to your private network (e.g., if you want 10.0.5.23, send it through the VPN).
    • Your AWS Node.js MCP server can now connect to mongodb://10.0.5.23:27017 directly.
    • That traffic flows securely over the VPN tunnel to your office.

Who sets it up?

  • In almost all companies, IT or DevOps sets up the VPN gateway.
  • As a programmer, you generally dont set up the network hardware, but once its in place, you can write code that uses those private IPs.

Typical steps for IT:

  • Create a Site-to-Site VPN in AWS VPC.
  • Configure your offices firewall/router to accept that VPN.
  • Add route tables so AWS knows 10.0.x.x is reachable via VPN.

2. SSH Tunnel Approach Port Forwarding to Mongo

The core idea

An SSH tunnel is like a one-lane secret pipe between AWS and a machine on your office network.
Instead of opening your whole LAN to AWS, you forward just one port (Mongos 27017) from AWS Local Mongo.

How AWS accesses Mongo through SSH

  1. You have some machine on your local network that can:

    • Connect to Mongo
    • Accept incoming SSH from AWS

  2. On the AWS server, you run:

    ssh -N -L 27017:localhost:27017 user@onprem-ip

    • -L 27017:localhost:27017 means:

      • When I connect to AWS localhost:27017, secretly forward it to onprem-ip:27017.

  3. Now your Node.js code on AWS connects to:

    MongoClient.connect("mongodb://localhost:27017");

    but that actually flows through the SSH tunnel into your office and reaches MongoDB.

Who sets it up?

  • You can set this up as a programmer (you just need SSH access to a machine inside your company network).

  • You dont necessarily need IT unless:

    • Firewalls block SSH
    • You need a dedicated always-on tunnel (then IT can automate it)

VPN vs SSH Tunnel Practical Differences

Feature VPN SSH Tunnel
Scope Whole network One port/service
Always-on Yes Needs to be maintained manually
Speed High (LAN-like) Slightly slower (one connection)
